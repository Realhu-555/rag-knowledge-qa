# Harness Engineering（驾驭工程）总结

> 来源：知乎《爆火的Harness Engineering》+ 什么值得买《万字讲透Agent Harness的十二大模块》+ 腾讯云《Harness Engineering：AI原生软件开发的未来范式》

## 一句话
**Agent = LLM + Harness**
Harness 是除了大模型本身之外，让 Agent 能真正干活的一切基础设施。

## 核心隐喻
大模型是**野马**，Harness Engineering 是**缰绳+马具+训练规范**。

## Harness 的 12 大核心模块

| 模块 | 说明 | 你项目里对应的 |
|------|------|--------------|
| 1. 状态管理 | 用状态机控制Agent行为流转 | ✅ 8状态+12事件+15规则 |
| 2. 工具调用 | Function Calling的完整生命周期 | 需要补 |
| 3. 记忆系统 | 短期/长期记忆管理 | ✅ Shared-Brain+Hindsight |
| 4. 规划推理 | Agent怎么拆解任务 | 需要补 |
| 5. 安全治理 | 权限管控、敏感操作拦截 | 需要补 |
| 6. 沙箱隔离 | 代码在隔离环境运行 | 了解但不深 |
| 7. 上下文管理 | 有限Token窗口内的信息优先级 | 需要补 |
| 8. 错误处理 | 重试、降级、熔断 | ✅ LLM重试1次+超时降级 |
| 9. 可观测性 | 追踪Agent每一步的决策 | ✅ Web展示中间产出 |
| 10. 成本控制 | Token预算、API配额 | ✅ 多模型切换省成本 |
| 11. 评估体系 | 怎么知道Agent好不好 | 需要补 |
| 12. 持续优化 | 数据回流改进系统 | 需要补 |

**你做到了4个（状态管理、记忆系统、错误处理、成本控制），另外4个有概念（工具调用、规划推理、可观测性、持续优化），4个比较缺（安全、沙箱、上下文、评估）。**

## 为什么需要 Harness Engineering（四大价值：R.E.S.T）

| 维度 | 解决的问题 | 具体手段 |
|------|-----------|---------|
| 可靠性 Reliability | Agent 随机翻车 | 检查点恢复、操作幂等性、行为一致性 |
| 效率 Efficiency | Token 消耗失控 | Token 预算管控、上下文压缩、资源配额 |
| 安全性 Security | 越权/数据泄露/指令注入 | 最小权限、沙箱隔离、输入输出过滤 |
| 可观测性 Traceability | 黑盒不可控 | 全链路追踪、决策归因、状态审计 |

## 核心架构：REPL 闭环容器

```
Read（感知管控）→ 上下文管理，结构化注入信息
     ↓
Eval（执行管控）→ 调用拦截器，合规校验+路由分发+异常监控
     ↓
Print（反馈管控）→ 结果结构化封装，注入上下文
     ↓
Loop（循环管控）→ 持续循环直至完成或终止
```

## 三大核心约束

1. **LLM 的非确定性** — 必须为每一次"不确定"设计降级路径
2. **有限上下文窗口** — 必须做状态分离+上下文压缩
3. **外部世界的无限状态** — 必须提供沙箱/API网关等工具

## 核心铁律：状态分离原则
LLM 必须视为**无状态计算单元**，所有跨轮次状态全部存于 Harness 管控的外部引擎。

## 架构分层

- **控制平面** — 决策与管控：任务调度、资源配额、行为规划、权限管控
- **数据平面** — 纯执行：Agent 运行实例、状态存储、记忆管理、沙箱环境

## 面试话术

> "Harness Engineering的核心思想是 **Agent = LLM + Harness**。它不是一个具体工具，而是一套工程体系——12大模块覆盖了Agent从开发到上线的方方面面。
>
> 我自己的AI Dev Platform项目里已经落地了其中几个：状态机管流程、共享记忆管信息、多模型切换管成本。但还有几个模块需要补，比如沙箱隔离和评估体系。
>
> 这个行业刚起步，Harness的成熟度就是Agent产品的核心竞争力。"
